# ProtoPilot HTML Showcase 内容规格

> 目标：把 HTML showcase 当作一套 slide-style 能力展陈来设计。结构必须先定义一级框架，再定义二级原型界面；每一个二级原型界面都对应一个独立 screen。本文档仅用于后续生成 showcase，不修改 skill 本身。

## 0. 设计原则

这不是一个单业务案例贯穿的产品原型，也不是工具说明文档。它应该像一套可以演示的产品发布 deck：每个二级页面都是一个可渲染、可定位、可 Spotlight、可被大改备注选中的 HTML 原型 screen。

核心原则：

- 一级框架负责讲解路径；二级原型界面负责具体页面。
- 二级原型界面必须独立成页，不允许写成“一页展示多个 xxx”。
- 不要把多个手机界面、多个 Web 界面或多个桌面界面塞进一个集合页。
- 每个二级页面只承载一个主原型或一个说明主题。
- 生成能力页以真实原型本体为主角，说明文字只能辅助，不要把原型包进说明卡片里。
- 宣讲解决方案和本地修订方案可以是 slide 式说明页，但仍然要符合 skill 的原生结构和标注规则。
- 不强行用一个业务场景串联，允许每个演示页使用不同场景，以展示丰富度。
- 第一屏做成网站 Hero，只保留品牌、slogan 和整体气质。

## 0.1 页面结构规则

为了避免再次生成“集合页”，所有页面按下面规则定义：

- 一级分类：只负责分组，例如“生成能力”“宣讲解决方案”。
- 二级页面：必须列出具体 screen，例如“音乐播放 App”“科技官网首页”。
- 每个二级页面必须写清楚：呈现形式、主界面、核心内容、要传达的点、备注。
- 如果一个二级页面里出现多个候选界面，必须拆成多个二级页面。
- 页面标题中不要使用“集合”“总览多个”“四宫格展示”等会诱导合并的表达。

推荐结构：

- 1 个 Hero 首屏。
- 8 个生成能力独立原型页。
- 4 个宣讲解决方案说明页。
- 2 个本地修订说明页。
- 总页数：15 页。

## 0.2 Skill 原生能力约束

Showcase 是 example，不是新 runtime。只能改 example 的内容包，不改 skill 本体。

必须遵守：

- 不改宣讲台壳层、PRD Viewer、目录、Spotlight、preview、改文字保存能力。
- 不重写 skill 原生 `.annotation` 视觉样式。
- 标注使用原生 `.annotation`，只允许在 example 内通过额外 class 控制位置。
- 标注必须放在主 surface 外侧，不进入 `.web-surface`、`.phone-screen`、`.app-screen` 或业务内容流。
- 长规则、背景、验收标准放 PRD Viewer 或正文说明；标注只放短句。
- 生成能力页中，真正的产品界面应是主 surface；不要让外层说明 slide 成为主 surface。

## 1. 开场

### 1.1 Hero 首屏

呈现形式：

- 网站式首屏。
- 整屏视觉，不使用手机框或普通卡片堆叠。
- 可以用深色、高级感背景，隐约出现多种原型界面的玻璃感碎片或能力信息。

主界面：

- 一个完整 Hero screen。
- 品牌、slogan 和整体气质是主角。

核心内容：

- 主标题：Finn ProtoPilot
- Slogan：Turn PRDs into review-ready prototypes.
- 中文辅助文案：把 PRD/spec 转成可宣讲、可本地修订、可质量检查的 HTML 原型交付包。
- 辅助标签：HTML Prototype / PRD Viewer / Presentation Shell / Local Revision

要传达的点：

- 这是一个完整、专业、有审美的原型生成 skill。
- 第一眼先建立品牌感和能力气质，不急着解释细节。

备注：

- 这一页只要 1.1。
- “你现在看到的是什么”并入宣讲解决方案章节。

## 2. 生成能力

这一组是 showcase 的核心。目标是展示 ProtoPilot 能生成多种形态，而不是只会做手机壳或后台表格。

重要规则：

- 这一组禁止集合页。
- 每个二级页面就是一个独立原型 screen。
- 说明文字不能压过原型本体。
- 生成的界面原型要视觉高级且界面完整，该有的部分都要有

### 2.1 音乐播放 App（参考网易云音乐）

呈现形式：

- 独立手机 App 原型页。
- 手机界面是主 surface。
- 说明文字可以放在旁边，但不能形成“说明卡片包住手机”的观感。

主界面：

- 播放器页面。

核心内容：

- 专辑封面。
- 歌曲名、歌手名。
- 播放进度。
- 播放控制。
- 歌词、推荐或队列入口。

要传达的点：

- ProtoPilot 能生成媒体类 App 的视觉氛围。
- 手机原型应有完整展示面积，可被单独 Spotlight。

### 2.2 小红书风格内容 App（参考小红书首页）

呈现形式：

- 独立手机 App 原型页。
- 手机界面是主 surface。
- 可在手机外侧放一条真实标注，用于演示短规则或待确认点。

主界面：

- 笔记信息流或图文详情页。

核心内容：

- 封面图。
- 标题、作者、正文摘要。
- 点赞、收藏、评论信息。
- 标签。

要传达的点：

- ProtoPilot 能生成内容社区类布局。
- 内容产品的信息密度和视觉气质应独立呈现。

备注：

- 标注必须使用 skill 原生 `.annotation` 样式。
- 标注放在手机外侧，不遮挡界面。

### 2.3 短视频 App（参考抖音）

呈现形式：

- 独立手机 App 原型页。
- 全屏视频界面是主 surface。

主界面：

- 抖音风格短视频播放页。

核心内容：

- 全屏视频区域。
- 右侧点赞、评论、分享操作。
- 底部作者、标题、音乐条。

要传达的点：

- ProtoPilot 能表达沉浸式移动端界面。
- 不同 App 类型不应共用同一种通用模板。

### 2.4 微信风格聊天 App（参考微信聊天列表）

呈现形式：

- 独立手机 App 原型页。
- 会话列表或聊天窗口是主 surface。

主界面：

- 通讯工具会话列表。

核心内容：

- 头像。
- 消息预览。
- 时间。
- 未读状态。
- 顶部搜索或新增入口。

要传达的点：

- ProtoPilot 能表达高频工具型 App 的信息密度。

### 2.5 SaaS 工作台

呈现形式：

- 独立 Web App 原型页。
- 浏览器/后台工作台是主 surface。

主界面：

- AI 内容审核台、CRM 线索管理或数据运营后台任选其一。

核心内容：

- 指标卡。
- 筛选或导航。
- 表格。
- 状态标签。
- 操作入口。

要传达的点：

- ProtoPilot 能生成工作台、后台、SaaS 类界面。
- Web 工作台不能和官网首页合并在同一页。

### 2.6 科技官网首页

呈现形式：

- 独立 Website 原型页。
- 官网首页本体是主 surface。

主界面：

- 科技产品官网 Hero。

核心内容：

- 产品名或品牌名。
- Hero 文案。
- CTA。
- 产品视觉或界面预览。
- 信任指标。

要传达的点：

- HTML showcase 不应全是功能型界面。
- ProtoPilot 也能生成营销官网、产品首页、发布页这一类非后台界面。

### 2.7 Mac App 原型

呈现形式：

- 独立 Mac App 原型页。
- macOS 软件窗口是主 surface。

主界面：

- 类似 Codex / AI coding assistant 的桌面应用。

核心内容：

- macOS 红黄绿窗口按钮。
- 左侧项目或会话 sidebar。
- 中间任务或代码对话区。
- 右侧 preview / diff / inspector 区域。
- 顶部 toolbar。

要传达的点：

- ProtoPilot 可以生成具有系统气质的桌面软件原型。
- Mac App 原型不等同于普通网页套壳。

### 2.8 Windows App 原型

呈现形式：

- 独立 Windows App 原型页。
- Windows 软件窗口是主 surface。

主界面：

- 类似 Codex / 企业 AI 助手 / 本地任务执行器。

核心内容：

- Windows 标题栏和窗口控制按钮。
- 左侧导航或项目树。
- 顶部命令栏 / toolbar。
- 中间任务日志、执行结果、diff 或文件列表。
- 底部状态栏。

要传达的点：

- ProtoPilot 不只生成移动端和 Web，也能表达桌面软件。
- Windows App 和 Mac App 应有不同系统气质。

## 3. 宣讲解决方案

这一组不是“验收能力”，而是把 ProtoPilot 作为一套宣讲解决方案来呈现。内容可以是 slide 式说明块、简单位置指示图、框架图，不需要手机演示。

### 3.1 原型交付包是什么

呈现形式：

- 左侧说明块 + 右侧简洁结构示意图。
- 不使用手机演示。

核心内容：

- PRD/spec 是需求来源。
- 原型界面是可讲解的生成结果。
- 宣讲台负责目录、PRD Viewer、标注、Spotlight、轻交互。
- 本地 preview 支持查看、检查和修订。

结构示意：

- PRD/spec → Plan → Prototype Screens → Presentation Shell → Local Review

要传达的点：

- 用户看到的不只是页面，而是一套可宣讲的原型交付包。

### 3.2 目录是讲解路线

呈现形式：

- Slide 式路线图。
- 可以用左侧目录栏示意 + 右侧路线卡片。

核心内容：

- 开场。
- 生成能力。
- 宣讲解决方案。
- 本地修订方案。

要传达的点：

- 目录不是网站导航，而是讲解路线。
- 多个独立原型界面不会散掉，使用者可以按章节讲。

### 3.3 PRD Viewer 是需求依据

呈现形式：

- 简单位置指示图。
- 左侧原型区域，右侧 PRD Viewer 区域，用线条或高亮说明二者关系。
- 这一页必须真实使用一条原生 `.annotation`。

核心内容：

- PRD Viewer 承载长说明、规则、背景和验收标准。
- 原型 screen 承载用户真实能看到的界面。
- 二者并排帮助使用者解释“为什么生成成这样”。

要传达的点：

- 不把长说明塞进手机屏或业务界面。
- PRD Viewer 是宣讲方案的一部分，不是附属文件。

备注：

- 标注只写短句，例如：长说明归位：规则、背景和验收标准不要塞进 screen。
- 标注放在主 surface 外侧。

### 3.4 标注与 Spotlight 是讲解辅助

呈现形式：

- 一张 slide 上画出界面区域、标注区域、Spotlight 放大区域。
- 必须真实使用一条原生 `.annotation`。

核心内容：

- Annotation：短规则、风险、待确认点。
- Spotlight：聚焦当前界面，放大查看细节。
- 演讲光标：指向关键区域。

要传达的点：

- 标注不是装饰，是讲解辅助。
- Showcase 不能只介绍标注而不使用标注。
- Spotlight 适合讲细节，不改变原型内容本身。

备注：

- 位置图本身不要和真实标注重叠。
- 标注不要做成自定义蓝白样式，使用 skill 原生样式。

## 4. 本地修订方案

这一组用 slide 思路讲本地修订闭环，不需要手机演示。重点是让用户知道：本地启动 preview 后，可以直接通过宣讲台面板编辑草图画板。

### 4.1 面板直接编辑草图

呈现形式：

- 左侧说明块 + 右侧 preview 壳与 Excalidraw 编辑面板示意。

核心内容：

- 场景：本地 preview 验收时，发现草图文字、形状或布局需要微调。
- 操作：点击宣讲台右上角「编辑」→ 选中对应步骤的草图卡片 → 在 Excalidraw 面板内直接修改 → 点「保存」。
- 回流：保存写回 `prototype-excalidraw/boards/*.excalidraw`，刷新 preview 即可看到更新。

示例操作：

- 把按钮文案「提交审核」改成「提交给主管」。
- 微调标注框位置，或拖动流程节点对齐。

要传达的点：

- 草图可直接在 preview 面板里编辑，不必打开外部 Excalidraw 文件。
- 编辑发生在本地 preview，保存后即时生效。
- 适合文字微调与形状位置调整；大规模结构重做仍可通过 AI 下一轮生成。

### 4.2 质量检查概念

呈现形式：

- 简洁检查报告 slide。
- 只让大家建立概念，不需要堆命令细节。

核心内容：

- 内容包完整。
- 计划和页面一致。
- PRD 可读取。
- 目录可导航。
- 标注位置安全。
- 预览可打开。

要传达的点：

- ProtoPilot 生成物不是“画完就算”。
- 它有基本的交付检查闭环。

## 6. 生成注意事项

- 第一屏必须像网站首屏，品牌和 slogan 是主角。
- 生成能力组禁止集合页；每个二级页面必须是独立 screen。
- 不要写“一页展示 4 个手机 App”“一页展示两种 Web 能力”这类规格。
- 不要做 Mobile 多状态、Web Drawer、流程 Storyboard 单独页。
- Mac App 和 Windows App 各一页，类似 Codex 类型界面即可。
- 宣讲解决方案部分用说明块和位置指示图表达，不要再塞手机演示。
- 标注必须真实出现，且使用 skill 原生 `.annotation` 样式。
- 本地修订部分用说明块和流程图表达，不要写成会议中实时协作。
- 质量检查只建立概念，不需要命令教学。
- 不做适用边界单独页。

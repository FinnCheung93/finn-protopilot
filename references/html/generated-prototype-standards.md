# 业务原型生成标准

本文件是业务原型生成效果的长期调优入口，只约束 `.proto-generated-area` 里的内容，不约束宣讲台固定能力。宣讲台契约与检查见 `references/quality.md`。

## 目标边界

- 生成的是 PM 宣讲用需求原型，帮助讲清流程、页面状态、关键规则和协作分歧。
- 不把原型当最终 UI 设计稿，不生成生产前端代码，不默认做完整 App 状态机。
- 完整度优先于压缩状态：业务原型要像一组可讲页面，而不是把所有分支塞进一个可交互小 App。
- 单个 `.journey-step` 默认只承载一个主要界面；两个手机/两个页面要拆成两个步骤，否则目录和全屏放大都会失真。
- 默认先生成可讲、可迭代版本；复杂和歧义很强时才使用 `prototype-brief.md` 规划。
- 业务生成区可以持续调优；宣讲台稳定能力优先通过脚本和模板维护。

## 必读来源

生成或更新前必须按顺序读取能找到的来源：

1. 源 PRD/spec。
2. 从需求目录向上找到的最近 `{Product}/Design/design.md`。
3. 产品级 `Design/assets`、`Design/references` 中的参考图片和素材。
4. 需求级 `assets`、`references` 中的参考图片和素材。
5. 已存在的 `index.html`，作为当前视觉表达和人工编辑的来源。
6. `prototype-brief.md`，仅当复杂需求确实需要它作为规划辅助。

如果 `design.md` 存在，必须读取，不需要用户点名。只有找不到时，才说明缺失并使用默认视觉兜底。

参考图不能只“发现”。只要产品级或需求级参考图存在，生成前必须打开或抽样打开关键图片，提炼布局、密度、圆角、组件和弹层规则。交付时说明具体使用了哪些图和借鉴点；没有采用的图要说明原因。

PRD 疑似在原有界面上修改时，先询问是否提供原界面图。用户可跳过；如果提供了，必须以原界面图为结构基准复刻并修改，不把截图直接当整屏背景图。

## 视觉与信息表达

- 业务原型视觉优先跟随 `design.md`、项目参考图、需求参考图和真实产品截图。
- `prototype-base.css` 的默认冷钴蓝只是一组基础能力和兜底组件，不应覆盖业务原型自己的产品视觉。
- 生成多界面前，先建立本需求的产品级语义块和局部样式 token，例如顶栏、列表行、主按钮、地图块、统计卡、提示条、弹窗。后续界面只组合这些语义块，避免同一份 PRD 内视觉飘移。
- 同一份原型必须有明确 UI 基线：状态栏、标题栏、卡片、按钮、列表、弹窗和底部操作区要一致。默认状态栏为 `12:30 + 5G`；除非 PRD 或参考图明确要求，不要混用 LTE、5G 或空白状态栏。
- 弹窗状态必须复用同一页面底层结构，只叠加遮罩与弹窗；弹窗底下的界面内容、标题栏、状态栏不能和原页面不一致。
- Mobile App 使用手机框；Web Page 和 Desktop Page 直接呈现大屏页面，不套假设备框，除非 PRD 明确需要。
- 一屏讲一个核心状态或决策点；PRD 中的 S1/S2/S3/S4、加载、空态、错误、过期、有数据等状态默认拆成独立 `.journey-step`，不要折叠进一个手机内的 Tab/Segment。
- 只有真实产品页面本来就有 Tab/Segment 时，才在手机内使用分段控件；不能用分段控件代替多状态讲解页。
- 页面数量多时先归纳：6 个以上界面必须拆成 2-4 个场景组，避免目录和画布变成一串平铺页面。
- 长规则、长表格、复杂说明放在注解、旁注或总结区，不塞进手机小屏。
- 文字必须可读，布局不能重叠；固定格式元素要有稳定宽高、网格或纵横比约束。
- 手机原型的背景、底部操作区、弹窗遮罩必须被限制在手机屏幕内；不要让业务弹窗覆盖整个宣讲台画布。
- 手机界面必须是完整屏：`.phone-frame > .phone-screen > .app-screen` 填满手机内部，不要再套一个大圆角白色“半截屏”承载整页。
- 手机内按钮优先复刻产品和参考图风格；不要把宣讲台通用 `primary-btn` / `secondary-btn` 直接当业务按钮，除非没有任何设计参考。

## 组件、图标、素材

- 常见操作和状态图标优先使用 Lucide，可用 CDN。
- 不用 emoji 当正式 UI icon；Lucide 不可用时，用文字或 CSS 占位。
- 不用裸 `>`、`...`、`→` 代替正式操作图标；返回、更多、箭头、关闭、设置等优先用 Lucide。
- 真实素材优先来自产品级/需求级 `assets/` 和 `references/`。
- 没有真实素材时，可使用简洁占位、CSS 形状或非最终说明性图片；不要假装占位图是真实设计资产。

## 文案与标注

- PRD 是中文时，原型里的标题、按钮、说明、注解、状态文案也尽量中文。
- 注解用于解释需求规则、状态差异、风险点和讲解重点，不用于装饰。
- 页面内可见文案应尽量来自 PRD 或设计上下文；缺失时可补 PM 可讲的简短文案，但不要编造业务规则。
- 所有需要讲解或后续修改的关键元素都要有稳定 `data-proto-id` / `data-proto-label`，包括卡片、列表行、按钮、状态块、弹窗、空态、图表和标注。
- 不在画布顶部或步骤之间生成横跨页面的“资料说明灰条”。背景来源、参考 PRD 章节、非必要说明应放在交付说明、步骤标题或简短标注里。

## 轻交互

- 只做帮助讲需求的轻交互：状态切换、Toast/Dialog、展开收起、分支切换、选中高亮。
- 不默认实现完整数据流、复杂权限系统、后端逻辑或真实表单提交。
- 交互控件要服务讲解，不要把原型变成需要培训才能操作的工具。

## 细则入口

业务生成区的结构、平台表达、组件复用、圆角/间距、弹窗和反模式，继续读 `references/html/generated-area-rules.md`。

## 反模式

- 只参考 skill 示例，不读取项目 `design.md` 和参考图。
- 把 PRD 长表格完整塞进手机屏幕。
- 用统一冷钴蓝风格替代真实产品视觉。
- 用 emoji 充当正式图标。
- 把宣讲台按钮当成业务 UI 的一部分。
- 为了“像 App”而生成过度复杂的状态机。
- 把 S1/S3/S4/S2-b 等多个 PRD 状态压缩进一个手机内 Tab，而不是拆成多个可讲界面。
- 把两个不同手机界面放进同一个 `.journey-step`，但只给一个步骤标题。
- 在手机里放一个大圆角白色半截容器当主界面，导致手机屏幕不完整或圆角破坏。
- 短页面全部加 `.is-extended`，导致长页模式被滥用；只有内容确实超过一屏时才启用长页。
- 贴底模块、地图底部或底部操作区只圆上角、不圆下角。
- 同一份原型里状态栏、标题栏、按钮或弹窗样式每屏都不一样。
- 把标注夹在两个手机之间、放进手机主内容流，或覆盖导航栏/主要内容。
- 把 8-10 个页面不分组平铺到一个流程里。
- 把手机内弹窗写成 `position: fixed` 或放到 `.app-screen` 外面。
